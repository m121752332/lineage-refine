# Batch refine tool and purchasable inventory capacity, built entirely in the Vue glue layer

Three features land together (2026-07-02): purchasable inventory expansion via the Olin dialogue, a unified `+N Name` display format, and a batch refine tool (整批衝裝工具). All of them live in `lineage.html` — the Vue glue layer — with zero changes to `assets/js/nav3d.js` or `assets/js/scene3d.js`, so the logic/presentation split and the Node test suite are untouched by construction.

**Batch refine is a modal sub-mode of the existing scroll-hammer flow**, gated by an explicit checkbox in the inventory toolbar (default off). When the box is unchecked, the single-item path is byte-for-byte identical to before; when checked, clicking a weapon/armor scroll opens the batch window instead of arming the hammer. The probability engine is **reused, not duplicated**: `isSafe()` / `successRate()` in `lineage.html` remain the single source of truth for both single and batch refining (note: the in-game engine is this inline formula — `assets/data/refine_rates.json` is fetched only by `refine_table.html` for display). Batch differs from single refining only in economics: each attempt consumes one scroll of the triggering variant (the run halts early with an explanation when scrolls run out), and **no gold enhancement fee is charged** — a deliberate positioning of the tool as a simulation accelerator, per the confirmed spec.

Rules that shape the run loop: cursed scrolls are **excluded** from batch mode (their success *decreases* the level, so a "refine until target" loop could never terminate — they fall through to the normal single-item flow). Blessed scrolls are allowed; their 50% `+2` jump is clamped to the chosen target level. Items already at or above the target enter the run as instant passes with zero attempts. The target-level combobox is capped by the configured `maxEnhLevel`, matching single refining. The scheduler is a self-rescheduling `setTimeout` chain (one attempt per 500 ms tick, slot-machine highlight), not `setInterval` — no overlapping ticks, and stop conditions are trivial. The window cannot be closed while a run is in progress.

Selected items **leave the persisted inventory the moment the run starts**; survivors return to empty slots only when the player clicks 回到背包. A page refresh mid-run therefore loses the in-flight items. This is an accepted risk: it mirrors the destruction stakes of refining itself, and avoids inventing a recovery localStorage key plus the reconciliation logic it would demand.

Inventory capacity is stored as `lineage_player.invSize` (default 100, step +100 at 10,000,000 gold per purchase through Olin, hard cap 1000). It lives in `lineage_player` rather than `lineage_config` because capacity is **account progression bought with in-game currency**, not a user preference — `lineage_config` remains reserved for settings the player can freely flip.
